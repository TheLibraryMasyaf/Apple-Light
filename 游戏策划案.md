# 2D平台跳跃黑白像素风小游戏策划案

## 1. 项目概述

### 1.1 游戏名称（暂定）
**《苹果灯 / Apple Light》**

### 1.2 游戏类型
- 2D 横版平台跳跃
- 像素风
- 关卡制
- 黑白单色美术
- 以“明/暗双形态切换”为核心机制

### 1.3 核心一句话描述
玩家操控一个火柴人，在黑白像素世界中通过跳跃、切换灯光状态、利用明暗两种不同的平台结构，逐步抵达关底完成关卡。

### 1.4 核心体验目标
- 让玩家体验“同一关卡两种形态”的空间变化感
- 通过灯光开关创造策略性路线选择
- 保持操作简单，但关卡解法有层次
- 强调视觉辨识度与黑白像素风格的统一

---

## 2. 玩法核心

### 2.1 核心机制：明 / 暗双形态
每个关卡都存在两个状态：
- **明状态**：灯亮，世界处于“明面结构”
- **暗状态**：灯灭，世界处于“暗面结构”

在两个状态下：
- 部分平台只在明状态下出现
- 部分平台只在暗状态下出现
- 部分平台始终存在，不受灯光影响
- 开关灯的装置只放置在“固定平台”上，这些平台不受明暗变化影响，保证玩家总能有操作锚点

玩家需要根据当前场景，判断何时切换灯光，利用两套平台结构进行移动和跳跃，最终到达关底出口。

### 2.2 核心循环
1. 观察当前关卡的明/暗结构
2. 通过移动和跳跃前往固定平台上的灯开关
3. 切换灯光状态
4. 利用切换后的平台结构继续前进
5. 到达出口，进入下一关

### 2.3 关卡设计重点
- 每一关都至少让玩家经历一次“看见路线 → 切换灯 → 路线变化”的过程
- 关卡难度逐步提升：
  - 早期：单次切换即可通关
  - 中期：需要多次切换并反复回到固定平台
  - 后期：需要记忆明暗平台位置并规划切换顺序

## 3. 操作方式

### 3.1 基础操作
- **A**：向左移动
- **D**：向右移动
- **W**：向上 / 爬梯子
- **S**：向下 / 下梯子
- **空格键**：跳跃
- **F 键**：触发开关交互（与开关平台接触时按下）

### 3.2 操作说明
- 玩家通过 A / D 控制左右移动，空格键跳跃
- 开关交互需主动按 F 键触发（非自动接触触发）
- **前 5 关**：梯子为可选机制，不强制使用
- **后 5 关**：W / S 用于爬下梯子或其他交互

### 3.3 操作原则
- 操作必须简洁，易于上手
- 跳跃手感要准确
- 切换灯光不直接占用按键，改为通过关卡中的开关交互来触发，强化“平台上寻找开关”的玩法

---

## 4. 视觉与美术风格

### 4.1 总体风格
- 黑白像素风
- 强对比度
- 场景简洁，轮廓明确
- 不使用复杂彩色贴图
- 重点突出平台边缘、角色轮廓、灯光范围

### 4.2 角色设定
主角仅用一个简化火柴人表现：
- 头部：圆形或小方块
- 身体：一条竖线
- 四肢：简化线条
- 动作尽量通过像素化姿态变化表达

### 4.3 场景表现与主题配色
#### 明状态配色方案
- 背景：纯白（#FFFFFF）
- 固定平台：纯黑（#000000），白色边框
- 可站立明平台：纯黑（#000000），白色边框
- 火柴人：纯黑（#000000）
- 灯：发光状态，白色内部，白色边框，散发白色光晕

#### 暗状态配色方案
- 背景：纯黑（#000000）
- 固定平台：纯白（#FFFFFF），白色边框
- 可站立暗平台：纯白（#FFFFFF），白色边框
- 火柴人：纯白（#FFFFFF）
- 灯：熄灭状态，深灰内部，白色边框轮廓，无光晕

### 4.4 灯光表现
- 关卡中央或场景显眼位置放置一个"大灯"
- 明状态下灯发光，投射白色圆形光晕到场景
- 暗状态下灯熄灭，仅显示边框轮廓
- 灯光切换建议使用简单的明暗过渡动画，而不是瞬间闪烁
- 灯光范围可表现为一个圆形或扇形照明区域，增强主题感

## 5. 游戏规则

### 5.1 平台分类
关卡中的平台分为三类：

#### 5.1.1 固定平台
- 永远存在
- 颜色或轮廓不随灯光改变
- 灯开关只能设置在这类平台上
- 主要作用：作为玩家切换形态时的安全站点

#### 5.1.2 明平台
- 仅在明状态下显示并可站立
- 暗状态下消失或失去碰撞
- 作为明状态路线的一部分

#### 5.1.3 暗平台
- 仅在暗状态下显示并可站立
- 明状态下消失或失去碰撞
- 作为暗状态路线的一部分

### 5.2 灯的切换规则
- 玩家接触开关后可切换灯状态
- 切换后整个关卡的平台可见性和碰撞状态更新
- 开关切换**无冷却时间**
- 角色**不允许在空中切换灯光**，仅可在接触开关时切换
- 切换过程中角色不能被卡进消失平台内部
- 建议切换前做一次轻微位置保护，例如：若玩家站在即将消失的平台上，允许短暂缓冲或自动弹出到安全位置

### 5.3 失败与重试
- 无失败惩罚
- 关卡底部为一层固定平台，且为所有关卡的起点
- 玩家即使跌倒底部也能重新回到起点开始
- 起点处设置重置关卡按钮

### 5.4 通关条件
- 玩家到达关卡出口
- 触发后进入下一关
- 普通流程共 31 关，完成后显示结局画面
- 满足隐藏条件后可进入第 32 关（隐藏关）
- 隐藏关通关后发放最终奖励

## 6. 关卡设计思路

### 6.1 关卡结构原则
每一关建议由以下部分构成：
1. 起点
2. 第一段基础跳跃区
3. 第一次看到开关
4. 第一次灯光切换
5. 明/暗路线分歧
6. 第二次切换或回返
7. 终点出口

### 6.2 关卡难度分层

#### 第一阶段：教学关
目标：让玩家理解“灯光切换会改变平台”。
- 平台布局简单
- 开关与出口距离短
- 切换一次即可通关
- 让玩家感知两种形态的差异

#### 第二阶段：组合关
目标：让玩家学会“先到固定平台，再切换”。
- 增加上下层平台
- 引入需要来回切换的结构
- 某些路段只有在特定状态下可通行

#### 第三阶段：策略关
目标：让玩家规划路线。
- 明/暗平台分布更复杂
- 需要记住灯的位置
- 可能要求连续切换多次
- 固定平台成为节奏控制点

### 6.3 关卡节奏建议
建议每关都遵循：
- 先易后难
- 先观察后行动
- 每次切换后都能明显感受到地图变化
- 避免长时间无变化的直线跳跃

### 6.4 示例关卡结构
#### 关卡1：入门
- 起点平台 → 固定平台上的开关 → 明平台通道 → 出口

#### 关卡2：初次分岔
- 起点 → 明平台上跳跃 → 到达固定平台 → 切换为暗状态 → 暗平台通路 → 出口

#### 关卡3：回返切换
- 起点 → 暗平台推进 → 固定平台开关 → 回到明平台路线 → 终点

#### 关卡4：双层结构
- 上层平台在明状态可达，下层平台在暗状态可达
- 玩家需要在两个层次之间切换状态进行推进

### 6.5 关卡总量与阶段分配（已确认）
- 总关卡数：**32 关**（31 个普通关卡 + 1 个隐藏关卡）
- 关卡 1-10：教学与基础组合阶段，重点理解明暗平台切换
- 关卡 11-20：进阶组合阶段，引入梯子互动（第 11 关起）
- 关卡 21-31：策略挑战阶段，引入移动平台（第 21 关起）
- 关卡 32：隐藏关卡（满足隐藏条件后解锁）
- 每个关卡至少存在两种可行路径，且通关时关卡状态可为明或暗

### 6.6 隐藏关卡设计（已确认）
- 隐藏关地形极简：仅保留底部一层平台
- 玩家沿底部平台前进至终点即可完成隐藏关
- 隐藏关完成后获得“背景中的灯”作为奖励（用于结局展示）

## 7. UI 与信息展示

### 7.1 必要 UI
- 当前关卡编号
- 当前灯状态：明 / 暗
- 当前关卡计时
- 全程累计计时
- 简单重试提示
- 通关提示

### 7.2 UI 风格
- 同样采用黑白像素风
- 字体建议使用像素风字体或等宽风格字体
- 信息尽量少，不遮挡关卡视野

### 7.3 提示方式
- 初次进入关卡时出现一句简短引导
- 当玩家靠近开关时显示交互提示
- 通关时显示“进入下一关”提示

## 8. 音效与反馈

### 8.1 音效方向
- 跳跃音效：短促、清脆
- 落地音效：轻微像素碰撞感
- 开关灯音效：有明确的“啪嗒”切换感
- 通关音效：短提示音

### 8.2 视觉反馈
- 跳跃时火柴人姿态轻微变化
- 开关灯时全屏明暗过渡
- 出口触发时出现简单闪烁或高亮边框
- 隐藏关奖励结算时，背景灯进行高亮展示并作为通关奖励呈现

---

## 9. 游戏体验目标

### 9.1 玩家感受
- 角色小但跳跃明确
- 关卡虽简单但切换带来强烈空间变化
- 玩家通过观察明暗平台学会规划路线
- 每次成功切换都获得“解谜推进”的满足感

### 9.2 难度控制目标
- 不能因操作过于复杂而劝退
- 不能因平台过少而无趣
- 每关都要有明确目标
- 切换灯光必须是有意义的，而不是纯装饰

## 10. 技术实现建议（后续编写代码时参考）

### 10.1 基础实现方式
建议使用 HTML + CSS + JavaScript 实现：
- HTML：游戏画布、UI、按钮和提示层
- CSS：像素风格界面、黑白主题、缩放和布局
- JavaScript：角色控制、物理、碰撞检测、关卡切换、灯光状态管理、按键输入处理

### 10.2 画面实现建议
- 使用 Canvas 绘制所有像素元素
- 关卡视图为单屏显示（960x540），无需摄像头滚动
- 所有平台必须在一个窗口内呈现，确保玩家一目了然

### 10.3 核心数据结构建议
每一关可由以下数据描述：
- 地图宽高：960 x 540（单屏可视）
- 固定平台数组
- 明平台数组
- 暗平台数组
- 移动平台数组（第 21 关起启用）
- 梯子交互点数组（第 11 关起按需启用）
- 开关位置
- 出口位置
- 出生点位置
- 关卡明暗通关记录（用于隐藏关解锁判定）

### 10.4 状态系统建议
至少包含：
- `lightState`: 明 / 暗
- `playerState`: 站立、跳跃、下落、受保护等
- `levelIndex`: 当前关卡
- `levelTimer`: 当前关卡计时
- `totalTimer`: 全流程累计计时
- `nearSwitch`: 当前是否靠近开关（用于交互提示）

### 10.5 颜色与主题系统
- 根据 `lightState` 切换以下元素：
  - 背景色（白 ↔ 黑）
  - 平台色（黑 ↔ 白）
  - 火柴人色（黑 ↔ 白）
  - 灯的表现（发光状态 ↔ 熄灭轮廓）
- 所有颜色切换需保持高对比度和清晰可见性

## 11. 关卡制作规范

### 11.1 平台放置规范
- 固定平台必须能让玩家稳定接触到开关
- 开关不能放在会随灯消失的平台上
- 明平台和暗平台要尽量形成有逻辑的路线
- 每关至少有一处“切换后新出现的关键路径”
- 每关至少提供两条可行路线（可侧重明态路线或暗态路线）
- 关底前应允许玩家通过路径选择，使最终通关状态可落在明或暗任一形态

### 11.2 关卡可读性规范
- 玩家在第一次看到关卡时，应该能大致分辨哪里是固定平台，哪里会因灯切换而变化
- 明暗变化后要有足够的视觉反差
- 不建议把所有平台都做得太碎，避免看不清路线

### 11.3 跳跃体验规范
- 跳跃距离要与平台间距匹配
- 不允许大量“极限跳”作为唯一解法
- 允许少量高难度跳跃作为后期挑战，但不能破坏整体节奏

## 13. 当前确认事项

以下内容已确认并已同步到对应章节：
- 总关卡数：32（31 个普通关卡 + 1 个隐藏关卡）
- 无死亡惩罚
- 无检查点设计，跌落后可快速从起点继续
- 移动平台：第 21 关开始加入
- W / S：第 11 关开始加入梯子互动
- 灯光开关无冷却时间
- 不允许空中切换灯光
- 计时系统：每关计时 + 通关总计时
- 项目定位为 Demo，需兼容 Web 与微信小游戏移植
- 每关至少存在两种路径，且可在明/暗任一状态下完成关底通关
- 隐藏关解锁条件：前 31 关通关过程中记录的明暗顺序可组成“AppleLight”的摩尔斯电码
- 隐藏关结构与奖励：仅底部平台 + 通关奖励为背景中的灯
